# 程式開發規則

## 實作前

- 依任務規模調整分析深度：錯字、文案或明確的局部修正可直接實作。
- 涉及架構、公共 API、相依套件、跨模組行為或不明確需求時，檢查 `.csproj`、`package.json`、`go.mod` 等設定，確認語言、框架、執行環境與套件版本，並分析受影響範圍。
- 優先閱讀現有程式碼，確認是否已有可重用的工具、函式、元件、共用模組或相似實作。
- 只有在複雜度或風險需要時才先提出方案；簡單且低風險的修改直接執行。

## 設計與可讀性

- 合理運用 SOLID、低耦合與高內聚，優先保持簡單、一致、易測試及可維護，避免為套用原則而過度設計。
- 類別、模組與函式應有明確職責；複雜邏輯拆成名稱清楚的小型步驟。
- 避免降低可讀性的過度語法糖、深層巢狀、複雜鏈式呼叫及難以理解的單行程式碼。
- 命名與 Coding Style 應延續專案慣例，不因個人偏好調整。
- 具有業務意義、重複使用或可能調整的值，依用途使用具名常數、列舉、設定或適當資料結構；單次使用且意義明確的值不必過度抽取。
- 註解用於說明背景、限制、原因、相容性、安全考量及非直覺決策，優先解釋「為什麼」；能以命名或拆分改善時，不以註解掩蓋難讀程式碼。
- 修改程式碼時同步更新或移除失效註解，不保留無用程式碼、停用邏輯或大量註解掉的舊程式碼。
- 避免重複程式碼，但不得為消除少量重複而建立過度抽象。

## 相容性、相依性與日誌

- 不使用高於專案設定之語言、框架或執行環境版本的語法與 API；除非需求包含升級，否則維持既有相容範圍。
- 不自行引入套件、框架或相依性；確有必要時先說明原因及影響。
- 正式日誌與異常記錄沿用既有框架、格式與層級。
- 除錯期間可暫時加入標準輸出，但最終交付前應移除；標準輸出本來就是專案介面或慣例時不在此限。

## 修改與搜尋

- 局部修改使用精確差異、搜尋替換或工具支援的局部格式，避免因少量變更重寫大型檔案。
- 變更區塊必須完整可讀，套用後不得留下無法編譯或解析的 `// ...`、省略號、「其餘不變」等占位內容。
- 除非與問題直接相關，搜尋時排除 `bin/`、`obj/`、`node_modules/`、`dist/`、編譯產物、相依套件目錄、大型日誌及二進位檔案。
- 已取得足夠資訊後停止重複搜尋；資訊可能因修改、同步或外部更新而失效時才重新確認。

## 完成條件

- 確認結果符合需求、未破壞既有行為，且程式風格符合專案規範。
- 若變更影響使用方式、架構、公共介面或維護方式，依 `documents.md` 更新相關文件。
